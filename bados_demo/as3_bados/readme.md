# AS3 bados Demo

This folder contains an AS3 declaration and companion iRule used to simulate source-IP behavior for BADoS testing.

## What is included

- `bados.json`: Full AS3 declaration for tenant `bados` and application `bados_service`.
- `bados_xff_insert.irule.txt`: Human-readable iRule source that is embedded (base64) in the AS3 declaration.

## What the declaration deploys

- `Service_HTTP` virtual server on `192.168.57.82:80`
- Backend pool `ltm_demo_pool` with members:
  - `10.0.2.71:8000`
  - `10.0.2.72:8000`
  - `10.0.2.73:8000`
  - `10.0.2.74:8000`
- HTTP profile with `trustXFF: true`
- DoS profile (`DOS_Profile`) with automatic heavy URL protection and stress-based bad actor detection enabled
- WAF policy reference from remote URL (`base-policy.json`)
- Two data groups used by the iRule:
  - `good_ips_datagroup`
  - `bad_ips_datagroup`

## How the iRule behaves

On `CLIENT_ACCEPTED`, the iRule checks the client source IP against internal data groups:

- If client IP is in `good_ips_datagroup`, it generates a randomized "normal" XFF address range.
- If client IP is in `bad_ips_datagroup`, it generates a randomized "malicious" XFF address range.

On `HTTP_REQUEST`, it inserts the generated value into the `X-Forwarded-For` header and logs it.

## Deploy example

From a host with access to BIG-IP AS3:

```bash
curl -sku admin:admin \
  https://<bigip-mgmt>/mgmt/shared/appsvcs/declare \
  -H "Content-Type: application/json" \
  -X POST \
  -d @bados.json
```

## Basic validation

1. Send traffic to `192.168.57.82:80` from source IPs that match entries in `good_ips_datagroup` and `bad_ips_datagroup`.
2. Confirm the inserted `X-Forwarded-For` values differ between good and bad source groups.
3. Check BIG-IP logs for `X-Forwarded-For:` entries generated by the iRule.

## Notes

- The iRule text file is the readable source; the AS3 object uses its base64 form.
- If you change the iRule source, re-encode it and update `bados.json` before redeploying.
